# 走查結論:六角格地形實驗 → 改走「參數化類比地形」

日期:2026-06-15
狀態:已與 yazelin 拍板。本文件取代「六角格地圖」作為 AI 戰場編輯器的地形方向。

## 背景

red-cliffs-3d 被當成「AI 戰場編輯器」的走查場(編輯器本體未來落在另一個 repo)。第三輪把**結構/陣營**資料化成功(已合併 main);第四輪嘗試把**地形 + 移動**也資料化,做法是 ASCII 六角格 tilemap(`feat/hex-map` 分支)。本文件記錄這輪的結論。

## 做了什麼(feat/hex-map,未合併)

- `data/hexmap.json`(ASCII tile 圖)+ `data/terrain-types.json` → `HexGrid` 模組由 tile 推導連續高度場/上色;`ground()` 轉接。
- 結構所在格自動整平(貼地);單位移動改 `data/scene.json` 的 hex 序列;船首朝前/交戰側身;運鏡離崖修正;schema + validator 擴充。
- 初版 hexmap 由原 `ground()` 反推。

## 結論:這版 hex 地形不 ship,地形保留原版類比

走查驗證出粗 tile 地形在「這個密集、手工精調的場景」上是**視覺降級**,而且在這個無法看畫面的環境裡幾乎無法盲調收斂。具體踩到的牆:

1. **細節落不到格子上**:漢水(細短支流)在 26 單位格下幾乎消失;河岸這種「窄水 + 乾岸漸層」粗格做不出來。
2. **共格衝突**:加寬長江讓艦隊浮起來 → 江陵/夏口/烏林營等**河岸結構的格子變成水、泡進去**;收窄 → 船擱淺。原本平滑地形用漸層同時滿足,粗格不能。
3. **set-piece 與座標強耦合**:箭雨、火船撞陣、運鏡全寫死對準原始艦隊 marks;hex 吸附把艦隊挪位 → 箭射空水、火船撞不到曹軍、艦隊重疊。naval 需要**細座標**,不是粗 hex。
4. **盲調不收斂**:地形是美術活,需要人在迴圈看畫面;本環境截圖不可用,只能數值驗位置,改一個破一個。

## 拍板:編輯器藍圖(被這輪修正)

| 層 | 編輯器格式 | 狀態 |
|---|---|---|
| **地形** | **參數化類比**:保留連續/平滑生成器(漂亮),把它的**特徵當資料**——河道中心線、紅崖中心/大小/高、丘陵與沼澤區塊、漢水支流參數… AI 改參數、生成器照畫。**不用 tilemap**。 | 新方向(下一輪 spec) |
| **結構** | catalog(`data/structures.json` + schema + validator) | ✅ 已合併 main |
| **單位/移動** | 細座標資料(**非粗 hex**;naval set-piece 尤其需要精確位置) | 理解修正(本輪教訓) |
| **音訊** | Freesound→Gemini 代聽→人耳 海選管線(特效/配樂/旁白) | ✅ 已做 |
| **驗收** | 人/AI 視覺驗收;**地形類美術活必須人在迴圈看畫面**,純數值驗不足 | 本輪教訓 |

核心心法不變:**AI 產資料、人驗收 + 微修;renderer/生成器讀資料**。修正的是「地形資料的形狀」——**參數化類比 > 粗 tilemap**。

## 為什麼「參數化類比」同時滿足漂亮 + 可編輯

- **漂亮**:沿用原本手調的連續地形引擎(`ground()` 那類 FBM + 河道雕刻 + 高斯崖),畫質不掉。
- **可編輯**:把寫死在函式裡的特徵參數(河道控制點、崖的中心/sigma/高度、各區塊地貌)抽成 JSON。AI 寫一組參數 = 一張新戰場;人可微修單一參數。
- 比 tilemap 好在:連續生成器天生有任意尺度的漸層(細支流、窄水乾岸都行),不受格子解析度限制。
